# 第 4 章 内存优化

## 4.1 问题背景

PageRank 的核心开销来自两个部分：一是图结构本身的存储，二是每轮幂迭代时对概率向量的更新。若直接采用稠密邻接矩阵，设图有 `N` 个节点，则需要存储 `N × N` 个浮点数。对本作业数据集而言，节点数约为 `10000`，若使用 `float32` 存储完整矩阵，光矩阵本体就约为 `10000 × 10000 × 4 ≈ 400 MB`，已经远超题目要求的 `80 MB` 峰值内存限制。因此，本项目必须在数据结构层面先把空间复杂度从 `O(N^2)` 压到 `O(M + N)`，再进一步控制迭代过程中的活动工作集。

本章采用两层优化思路。第一层是 CSR（Compressed Sparse Row）稀疏表示，目标是只保存真实存在的边；第二层是按目标节点做真分块，把边先落盘，再用 `np.memmap` 在每轮按块读取，使 `r_new` 的活动更新片段始终限制在当前块内。最终的主提交版本采用 `CSR + Block Matrix` 组合，其中 CSR 负责压缩图结构，Block Matrix 负责限制迭代阶段的常驻内存。

## 4.2 CSR 的空间优势

本项目中，A 角色负责输出 CSR 三元组：

- `row_ptr`：长度为 `N + 1` 的行指针
- `col_idx`：长度为 `M` 的列索引
- `out_deg`：长度为 `N` 的出度数组

对于源节点 `u`，其所有出边位于区间：

```text
col_idx[row_ptr[u] : row_ptr[u + 1]]
```

这样做的好处是只保存真实边，不保存不存在的零元素。空间复杂度从稠密矩阵的 `O(N^2)` 下降为：

```text
O(N + M)
```

在本数据集上，`N = 10000`、`M = 150000`。如果索引统一使用 `int32`，则仅就核心结构估算：

- `row_ptr` 约 `(10001 × 4) ≈ 39 KB`
- `col_idx` 约 `(150000 × 4) ≈ 586 KB`
- `out_deg` 约 `(10000 × 4) ≈ 39 KB`

这意味着图结构本身只需约 `664 KB` 量级的原始数组存储，已经远低于稠密矩阵方案。更重要的是，CSR 让 PageRank 的每轮迭代不再需要访问整张 `N × N` 矩阵，而只需顺着真实边做传播，从而避免了大量无意义的零乘加。

## 4.3 分块策略：为什么不是普通切片

仅有 CSR 还不够。若直接在内存中保留完整的 `r_new`、反向邻接关系和辅助临时数组，Python 解释器、NumPy 运行时和若干中间缓冲区叠加后，仍可能让 RSS 峰值接近题目上限。因此，本项目的第二层优化不是“对数组做切片”，而是真分块：

1. 先按**目标节点区间**把边分桶。
2. 每个桶落盘为一个独立的 `.bin` 文件。
3. 每轮迭代时，只 `memmap` 当前块的边记录。
4. `r_new[node_start:node_stop]` 作为当前唯一活动更新片段。

这里按目标节点分块，而不是按源节点分块，原因是 PageRank 更新写入的是目标节点项 `r_new[v]`。若按目标节点分块，则一个块内的所有边贡献都写回同一个局部连续区间，可以避免在多块之间频繁来回写入。

本项目的块文件记录格式为：

```text
(src: int32, local_dst: int32)
```

其中 `local_dst = dst - node_start`。这样在迭代阶段无需重复做全局下标减法，直接对块内局部数组累加即可。分块构建一次完成，后续每轮复用。

## 4.4 memmap 与活动工作集控制

在分块迭代阶段，完整的 `r` 与 `r_new` 不再作为普通 `ndarray` 常驻，而是映射到临时文件：

```text
r      -> rank_a.bin
r_new  -> rank_b.bin
```

这样，操作系统只会把当前访问到的页调入内存。算法本身在代码层面遵循以下约束：

- `r_new` 只通过当前块视图 `r_new[node_start:node_stop]` 写入
- 边记录通过 `np.memmap` 顺序读取
- 块内贡献使用固定大小缓冲区处理，避免按边数生成大临时数组
- 每轮结束后只交换 `r` 与 `r_new` 的角色，不复制整向量

对应的块迭代公式仍然保持与普通 PageRank 一致：

```text
r_new[v] = (1 - beta) / N
         + beta * dangling_mass / N
         + beta * sum(r[u] / out_deg[u] for u -> v)
```

不同之处仅在于“对每个 `v` 的更新”被组织为“对每个目标块的更新”。因此，分块不会改变数值结果，只改变访问顺序与内存布局。

```text
伪代码：
for iter in range(max_iter):
    dangling_mass = sum(r[u] for u in dead_ends)
    base = (1 - beta) / N + beta * dangling_mass / N
    for each block:
        block_view.fill(base)
        for each edge (src, local_dst) in block_file:
            block_view[local_dst] += beta * r[src] / out_deg[src]
    check L1 convergence
    swap(r, r_new)
```

## 4.5 float32 的选择

所有概率向量默认采用 `float32`，原因有三点。

第一，PageRank 本身是迭代近似问题，题目规定的停止阈值为 `1e-8`，并不要求达到双精度线性代数求解的极限精度。第二，`float32` 可将向量、缓冲区和临时块视图的空间消耗直接减半。第三，本项目的最终评价重点在于时间与 RSS 峰值，而不是极限数值精度。

从实验 E7 的结果看，`float32` 与 `float64` 在 Top-10 签名上保持一致，说明排序层面没有发生可见偏差。以 `experiments/report_summary.json` 的最终汇总结果为准：

| dtype | 平均峰值 RSS / MB | 平均时间 / s | 迭代轮数 |
| --- | ---: | ---: | ---: |
| `float32` | `34.22` | `0.789` | `15` |
| `float64` | `35.34` | `0.719` | `12` |

数据源：`experiments/report_summary.json`。

这里 `float64` 的耗时略低，主要原因不是单次运算更快，而是双精度下收敛轮数减少。尽管如此，主提交版本仍建议固定为 `float32`，因为题目明确把内存控制作为核心约束，而 `float32` 在更大数据规模上会保留更稳妥的安全余量。

## 4.6 实测结果与分析

基于 Windows 开发环境、`Data.txt` 数据集、`beta = 0.85`、`eps = 1e-8`，实验 E3 表明块数 `K` 会在“块内数组大小”和“块切换开销”之间形成权衡：

| K | 平均峰值 RSS / MB | 平均时间 / s | 迭代轮数 |
| ---: | ---: | ---: | ---: |
| `4` | `35.09` | `0.704` | `15` |
| `8` | `34.98` | `0.729` | `15` |
| `16` | `34.92` | `0.856` | `15` |

数据源：`experiments/report_summary.json`。

可以看到，本数据规模下继续增大 `K` 并不会显著降低 RSS，反而增加了块切换和文件访问开销。原因在于 `N = 10000`、`M = 150000` 本身并不大，真正占 RSS 的并不只是图数据，还包括 Python 解释器、NumPy 运行时和子进程开销。因此，过细分块的收益有限。

E4 对比纯 CSR 与 `CSR + Block` 的结果如下：

| 模式 | 平均峰值 RSS / MB | 平均时间 / s | 迭代轮数 |
| --- | ---: | ---: | ---: |
| `csr` | `34.36` | `0.726` | `15` |
| `csr_block` | `34.66` | `0.838` | `15` |

数据源：`experiments/report_summary.json`。

当前最终汇总结果中，分块版本的 RSS 与纯 CSR 接近，时间略高。这说明在本数据规模上，分块更多体现为“工程上可扩展的内存控制手段”，而不是当前小规模数据下的绝对内存最优。若数据集扩大，分块的收益会明显放大，因为其活动工作集不再随全部边数线性增长。

## 4.7 本章结论

本项目的内存优化不是单点技巧，而是数据结构与执行策略的组合设计。CSR 解决“图怎么存”的问题，把空间复杂度从 `O(N^2)` 压到 `O(N + M)`；分块 + `memmap` 解决“迭代时怎么访问”的问题，把整轮更新的活动工作集限制在当前块；`float32` 则进一步为主提交版本提供了更稳妥的内存余量。最终，在当前数据集上，主提交版本 `csr_block + float32 + K=8` 的平均峰值 RSS 保持在约 `34.66 MB`，明显低于作业要求的 `80 MB` 上限，也为最终提交提供了明确的工程基线。
