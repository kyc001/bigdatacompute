# 第 7 章 总结与反思

## 7.1 工作总结

本项目围绕 PageRank 在一万节点、十五万条边的有向图上的计算展开。作业不仅要求得到 Top-100 结果，还要求在 Python 实现下控制内存峰值与运行时间，并强制实现 Sparse Matrix 与 Block Matrix 优化。团队因此采用接口先行的协作方式，由 `INTERFACE.md` 冻结输入输出、dead-end 公式、CLI 和 stdout JSON，再由 A、B、C 分别推进算法、工程优化和实验报告。

当前阶段已经完成 A 侧核心算法、B 侧分块实现与打包配置、C 侧数据分析与报告统稿草稿。实验显示，主提交模式 `csr_block + K=8 + float32` 能在当前数据集上稳定完成计算，内存峰值远低于作业要求的 `80 MB` 上限。

## 7.3 经验与反思

第一，接口冻结比临时口头约定更可靠。`sweep.py` 只需要遵守 `main.py` 的 CLI 和 stdout JSON，就可以在不 import 主程序的情况下完成实验扫描。这降低了实验脚本对内部实现的耦合，也避免内存测量被同进程导入污染。

第二，内存优化不能只看数组大小。当前图结构本身并不大，真正影响 RSS 的还包括 Python 解释器、NumPy 运行时、临时缓冲区和子进程采样方式。因此报告中应区分“理论数组空间”和“实测 RSS”，不能用估算值替代 benchmark 数据。

第三，图表必须保持可追溯。C 侧所有图都从 CSV/JSON 生成，统一以 `experiments/report_summary.json` 作为最终数字来源。对于接口没有提供的逐轮 residual 历史，不用估计值填充，而是在报告中说明当前只能展示最终残差汇总。
