# 第 1 章 任务背景与数据集

## 1.1 任务背景

本次作业要求在给定有向图数据集 `Data.txt` 上计算 PageRank 分数，并输出排名前 100 的节点及其分数到 `Res.txt`。输入文件每一行包含两个整数 `FromNodeID ToNodeID`，表示一条从源节点指向目标节点的有向边。作业明确要求程序需要迭代至收敛，必须处理 dead-end 与 spider-trap，并且至少报告阻尼系数 `beta = 0.85` 时的 Top-100 结果。

从工程约束看，本题不仅是 PageRank 正确性问题，也是一道内存优化题。作业要求 Python 入口文件为 `main.py`，运行峰值内存最好低于 `80 MB`，运行时间低于 `60 s`，并强制实现 Sparse Matrix 与 Block Matrix 优化。团队因此采用 CSR 稀疏存储压缩图结构，再用按目标节点分块和 `memmap` 控制迭代阶段的活动工作集。

## 1.2 数据集规模

使用 `analyze_dataset.py` 对 `Data.txt` 做一次完整扫描，得到如下统计结果：

| 指标 | 数值 |
| --- | ---: |
| 节点数 `N` | `10000` |
| 边数 `M` | `150000` |
| 平均出度 | `15.00` |
| 平均入度 | `15.00` |
| 最大出度 | `36` |
| 最大入度 | `32` |
| 最大总度数 | `57` |
| dead-end 节点数 | `1500` |
| dead-end 占比 | `15.00%` |
| 弱连通分量数 | `501` |
| 最大弱连通分量节点数 | `9500` |
| 最大弱连通分量占比 | `95.00%` |
| 孤立节点数 | `500` |
| 自环数 | `15` |
| 重复边数 | `0` |

从规模上看，本数据集节点数为一万，边数为十五万，图整体是稀疏图。若使用稠密邻接矩阵，单是 `10000 × 10000` 的 `float32` 矩阵就需要约 `381 MB` 以上，不可能满足内存限制；而 CSR 只需要保存真实存在的边和行指针，核心数组规模可以压到 MB 级以内。这也是团队选择 CSR 作为主存储结构的直接原因。

## 1.3 度分布与 dead-end 特征

度分布图见 `report/fig/degree_distribution.png`。从统计结果可以看到，平均入度和平均出度都等于 `15`，这是由总边数与节点数的比例决定的。但最大出度为 `36`、最大入度为 `32`，说明节点连接并不完全均匀，少数节点拥有更高的链接集中度。

更重要的是，数据集中存在 `1500` 个 dead-end 节点，占全部节点的 `15.00%`。这些节点没有出边，如果在迭代中直接忽略，会导致概率质量逐渐流失，最终 PageRank 向量不再保持归一化。团队在 `INTERFACE.md` 中冻结了 dead-end 质量补偿公式：每一轮把 dangling mass 按 `beta * dangling_mass / N` 均分回所有节点，再叠加随机跳转项 `(1 - beta) / N`。这部分处理是结果正确性的必要条件。

## 1.4 连通性观察

按无向图视角统计弱连通分量，数据集中共有 `501` 个弱连通分量，其中最大弱连通分量包含 `9500` 个节点，占总节点数 `95.00%`。同时存在 `500` 个孤立节点。这个结构说明图的主体部分高度集中，但也存在一批完全孤立或与主体无连接的小分量。

对 PageRank 而言，弱连通分量较多本身并不会导致算法失效，因为 teleport 机制会让随机游走在全图范围内保持可达性；但它会影响低排名节点的分数分布，也让 dead-end 补偿更重要。报告后续实验分析中，Top-10 排名主要受主体连通分量中的高入度或高质量来源节点影响。

## 1.5 本章小结

本数据集是一个典型稀疏有向图：`N = 10000`、`M = 150000`、平均度为 `15`。CSR 能显著降低图存储成本，Block Matrix 则用于进一步控制迭代过程中的内存活动范围。数据集中 `15.00%` 的 dead-end 节点和 `501` 个弱连通分量说明，正确处理 dangling mass 与 teleport 不是附加细节，而是 PageRank 收敛和结果可信的基础。
