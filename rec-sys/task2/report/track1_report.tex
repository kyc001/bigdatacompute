\begin{abstract}
Track1 要求在预训练矩阵分解模型接口上利用增量评分降低测试 RMSE，并尽可能压缩 \texttt{update()} 与预测的总耗时。完整增量 SVD 需要反复读写 1024 维用户和物品向量，预测阶段也会被高维点积主导；本实验转而采用低参数残差校准：增量阶段统计相对全局均值的用户/物品残差，使用计数形状函数和用户分段先验稳定长尾估计，预测阶段只执行两表相加与评分截断。最终提交实现使用 128 个浮点校准参数，所有实验均在当前本地 Docker 环境中重新测量；主要方法对比固定重复 5 次，线程消融每组重复 10 次并按稳健规则剔除异常值。统一方法对比中，最终实现将 RMSE 从 1.023239 降至 0.918947，5-run 总耗时为 0.025430 s。本文重新设计实验展示：表格用于说明数据规模、参数预算、实验问题和候选方案职责；图分别解释优化路径、Pareto 权衡、分段耗时、采样强度、组件贡献、线程敏感性和复杂度来源，避免图表表达同一组信息。
\end{abstract}

\section{任务与约束}

本任务使用 MovieLens 20M 评分数据\cite{harper2015movielens}。评测系统提供预训练用户矩阵 $P$、物品矩阵 $Q$ 和全局均值 $\mu$，标准矩阵分解预测为
\begin{equation}
\hat r_{ui}=\mu+P_u^\top Q_i .
\label{eq:svd}
\end{equation}
矩阵分解是推荐系统中的经典方法\cite{koren2009matrix}，但本题不是单纯追求模型复杂度，而是在 RMSE 有效改善后按运行时间竞争。预测输出必须截断到 $[0.5,5.0]$，越界用户或物品返回全局均值。

\section{模型设计}

\subsection{从高维更新退到残差统计}

早期尝试包括完整点积残差、截断 SVD 预测和少量 P/Q 头部特征。它们验证了一个关键现象：本地增量数据中最强的可利用信号是用户和物品的系统性偏差，而不是高维交互项的细粒度重训。因此最终模型以全局均值为基准，定义增量残差
\begin{equation}
e_t=r_t-\mu .
\label{eq:residual}
\end{equation}
物品侧每隔 $s_i$ 条样本采一次，用户侧每隔 $s_u$ 条样本采一次；最终提交使用 $s_i=4,s_u=10$。对被采样样本累计
\begin{equation}
S_i=\sum_{t\in\mathcal I_i}s_i e_t,\quad C_i=\sum_{t\in\mathcal I_i}s_i,\qquad
S_u=\sum_{t\in\mathcal U_u}e_t,\quad C_u=|\mathcal U_u|.
\label{eq:stats}
\end{equation}
这里 $\mathcal I_i$ 和 $\mathcal U_u$ 表示采样后落到物品 $i$ 或用户 $u$ 的样本集合。物品侧残差乘以 $s_i$，用于保持采样统计量的量级。

\subsection{低参数校准函数}

最终预测由用户分数和物品分数相加得到：
\begin{equation}
\hat r_{ui}=\operatorname{clip}\left(g_u(S_u,C_u)+h_i(S_i,C_i)\right).
\label{eq:pred}
\end{equation}
用户侧和物品侧的校准函数为
\begin{align}
g_u(S,C)&=p(u)+a_1\log(1+C)+a_3(C+1)^{-1/2}+a_5\frac{S}{C+\lambda_u},\\
h_i(S,C)&=a_2\log(1+C)+a_4(C+1)^{-1/2}+a_6\frac{S}{C+\lambda_i}.
\end{align}
其中 $\lambda_u=20,\lambda_i=5$。可学习浮点参数包括 7 个全局系数、119 个用户分段值和 2 个 shrinkage 常数，共 128 个。分段边界是整数索引映射，用于把用户快速映射到 119 个区间，不作为浮点校准参数。这样的参数含义是计数形状、残差均值和用户群体先验，而不是本地测试样本记忆。

\Cref{tab:param} 是模型设计与大参数拟合的分界线。若直接给每个用户或物品分配参数，本地 RMSE 可能更低，但参数会快速变成测试集 ID 记忆；当前设计只学习跨用户群体和计数形状的映射，因此隐藏 batch 中只要用户、物品分布没有完全改变，模型仍然能按残差统计规律工作。这个限制也迫使优化重点从“增加参数”转向“减少高频路径成本”。

\section{实现细节}

最终代码中的预测函数很短，核心是边界检查、更新前保护、两表相加和 clip：
\begin{lstlisting}[language=C++]
float predict(int user_id, int item_id) {
    if (static_cast<unsigned>(user_id) >= static_cast<unsigned>(users) ||
        static_cast<unsigned>(item_id) >= static_cast<unsigned>(items)) {
        return global_mean;
    }
    if (!has_updates) {
        return global_mean;
    }
    if (use_segment_model) {
        return clip_score(user_score_data[user_id] + item_score_data[item_id]);
    }
    return clip_score(global_mean + user_score_data[user_id] + item_score_data[item_id]);
}
\end{lstlisting}

增量阶段按全局偏移确定采样相位，分别扫描物品侧和用户侧。采样是确定性的，不依赖随机数，也不需要额外状态：
\begin{lstlisting}[language=C++]
const int user_start = (user_stride - base_user_phase) % user_stride;
const int item_start = (item_phase + item_stride - base_item_phase) % item_stride;
for (int idx = item_start; idx < n; idx += item_stride) {
    const Rating& r = ratings[idx];
    apply_item_update(r.item, r.rating - mean);
}
for (int idx = user_start; idx < n; idx += user_stride) {
    const Rating& r = ratings[idx];
    apply_user_update(r.user, r.rating - mean);
}
\end{lstlisting}

计数形状项中的 $\log(1+C)$ 和 $(C+1)^{-1/2}$ 在加载阶段预先写入查表。绝大多数更新只需数组访问和乘加，超过查表范围时才直接计算。线程设置根据本地每线程 10 次稳健消融同步为 4 线程，报告中的线程实验用于说明该选择及其环境敏感性。

实现上有三个容易被忽略的边界条件。第一，模型在没有收到任何增量更新前返回全局均值，避免加载阶段的用户先验影响“更新前”行为。第二，隐藏数据若不满足本地用户数和物品数，代码会走保守 fallback 路径，而不是错误访问本地维度假设下的数组。第三，评分截断仍在最终预测路径中显式执行，避免少量残差偏置把预测推出题面要求的 $[0.5,5.0]$。这些逻辑本身会带来少量分支，但比隐藏评测中出现无效提交更重要。

内存布局也按热路径重新整理。用户和物品的累计量只保留 \texttt{sum} 与 \texttt{count}，最终预测所需的值提前刷新到 \texttt{user\_score} 与 \texttt{item\_score} 两个连续数组中。这样并行 RMSE 预测时不需要访问结构体数组、计数表或对数函数。早期版本尝试过在预测阶段检测脏标记并重建分数表，这会让某些重复评测更快，但实质上把 \texttt{update()} 工作推迟到了 \texttt{predict()}；最终报告中的方案明确放弃该做法。

\subsection{最终方案选择原则}

最终方案不是简单选“最快点”或“最低 RMSE 点”。本实验按四层规则筛选候选：首先必须通过有效性检查，预测前行为、越界 fallback 和评分截断都不能破坏题面接口；其次 RMSE 要有足够余量，避免模型在隐藏 batch 上因为分布轻微变化而贴近无效边界；第三才比较端到端 5-run 总耗时，并且只接受来自 \texttt{update()} 内真实计算减少的收益；最后检查实现是否存在延迟重建、跨轮缓存或把更新工作转移到 \texttt{predict()} 的行为。

  \end{tabular}

\Cref{tab:select} 给出了本次最终替换的依据。旧 stride=8 版本曾满足实现纪律和有效性，但在同一测量口径下已经不再是最优折中；stride=4 在保持更新机制不变的情况下增加物品侧有效样本，降低 RMSE，同时没有引入预测阶段补更新。因此最终源码应当使用 stride=4，而不是把它停留在候选或消融结果中。

\section{候选架构权衡}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.86\linewidth]{pareto_frontier.png}
  \caption{候选架构的本地 Pareto 分布。虚线连接当前重测中不被更快且更准方案支配的点。}
  \label{fig:pareto}
\end{figure}

\Cref{fig:pareto} 显示了比表格更完整的取舍关系。灰色消融点说明去掉关键组件后即使运行时间短，RMSE 也会退化；完整物品统计和触达刷新精度好，但时间更高；stride=16 很快但接近 0.93 边界。最终 stride=4 查表采样位于当前本地 Pareto 前沿：它不如完整物品统计精确，但明显更快；它不如 stride=16 极限快，但 RMSE 安全余量更大。

这一结论也解释了为什么最终源码必须跟随实验结果更新。旧的 stride=8 版本曾作为保守提交点，但在同一 Docker 环境重新测量后，stride=4 在 RMSE 和端到端时间上都更合适；因此本报告把 stride=4 提升为正式方案，而不是继续把它作为候选点展示。性能优化报告需要让最终实现与实验结论一致，否则优化路径就只是一组记录，不能支撑提交决策。

\Cref{fig:breakdown} 解释了速度优化发生在哪里。完整物品统计的 5-run update 耗时为 0.033956 s，而最终 stride=4 的 update 耗时降到 0.016383 s；预测段本身只有毫秒级，差异较小。因此进一步优化速度时，优先减少 update 写入和刷新，而不是在 predict 中继续做细枝末节优化。

\section{线程与复杂度}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.78\linewidth]{thread_scaling.png}
  \caption{线程数消融。纵轴是相对 1 线程的速度，点旁标注为 10 次测量并剔除异常值后的单轮均值。}
  \label{fig:thread}
\end{figure}

\Cref{fig:thread} 说明线程数不是单调收益。当前本地容器中 4 线程最快，8 和 16 线程反而变慢；本轮 10 次测量中 MAD/IQR 规则没有剔除任何点，说明结论不是单个异常值造成的。原因是最终预测路径只剩少量数组访问和加法，OpenMP 调度开销、缓存行为和容器调度会主导短循环。因此最终源码默认使用 4 线程；若远端机器表现不同，可以通过编译期线程参数调整，而不改变算法。

\begin{figure}[H]
  \centering
  \includegraphics[width=0.80\linewidth]{complexity_sources.png}
  \caption{复杂度来源对比。横轴为近似操作或写入次数，对数坐标。}
  \label{fig:complexity}
\end{figure}

\Cref{fig:complexity} 把前面实验的原因归结为高频路径减少。预测阶段从完整点积的约 2048 次相关操作降到 3 次左右；物品写入从每 100 条样本写 100 次降到约 25 次；计数函数通过查表从高频计算变成常数访问。速度收益来自这些路径变化，而不是单纯把系数调小或调大。

\section{不保留方案与风险}

完整 SVD 更新没有作为最终方案保留。它最贴近题面中的矩阵分解公式，但每条样本读写两条 1024 维向量，在本地规模下很快变成内存带宽问题。截断点积和少量 P/Q 头部特征也被测试过，它们能提供一定精度收益，但预测阶段会重新引入乘加循环。与最终残差表相比，这类方法的主要问题不是 RMSE，而是每次测试预测都要付出额外成本。

延迟重建和跨轮缓存也没有保留。它们在固定本地 runner 上可能让重复评测的总耗时更低，但依赖“多轮评测在同一进程中重复相同数据”的状态结构，不属于稳健的增量算法。最终版本要求每次 \texttt{update()} 后分数表已经可用，\texttt{predict()} 不触发重建、不进入临界区、不复用上一轮测试结果。这样会牺牲一些本地极限速度，但隐藏评测中更可靠。

大参数模型同样没有保留。早期训练过更大的线性头、MLP 或逐 ID 近似表，它们在本地测试上可能降低 RMSE，但参数含义容易退化为本地测试标签拟合。当前 128 个浮点参数的设计更窄：系数只控制计数形状、残差 shrinkage 和用户分段先验。这个约束限制了最低 RMSE，但降低了隐藏批次失效的风险。

剩余风险主要有两点。第一，本地 Docker 的短时间测量仍有调度噪声，线程消融中 4 线程最快不代表远端也一定最快；本次按当前统一测量口径将默认线程同步为 4。第二，最终 stride=4 比完整物品统计略损失 RMSE，换来明显更低的 update 成本；若后续目标从速度转向更低 RMSE，应优先研究如何保留更多物品统计而不增加写入热点，而不是继续扫细小权重。

\section{结论}

本实验最终采用在线残差校准而不是完整增量 SVD。实验过程表明：增量评分中最强信号是用户/物品残差；低参数计数形状和用户分段先验能稳定长尾估计；真正的速度瓶颈在 update 写入和刷新，而不是预测中的几次加法。当前本地 Docker 统一方法对比中，最终实现将 RMSE 从 1.023239 降至 0.918947，5-run 总耗时为 0.025430 s。重新设计后的图表展示了完整优化过程和架构取舍：完整物品统计是精度上界，stride=16 是速度极限但精度风险较高，最终 stride=4 查表采样是当前本地最优折中并已写入正式源码。后续若继续优化，优先方向应是减少 update 阶段写入，同时保持 stride=4 的 RMSE 余量，而不是恢复完整 1024 维点积。
